# Quick 260512-m02: Real Timer Fix (Interval 8ms + lastFrameTime Accumulator) Summary

**One-liner:** Fixed 22fps playback drift by lowering `timerPlayback.Interval` to 8ms and switching `lastFrameTime` update to ideal-frame accumulator, eliminating per-tick 14ms time loss and OS scheduler-induced WM_TIMER rounding (33→45ms).

## Background

Phase chain: `260512-ifn` (jitter instrumentation) → `260512-kma` (timeBeginPeriod 1ms scheduler) → **260512-m02** (real timer fix).

After `260512-kma` applied `timeBeginPeriod(1)`, PerfLog measurement on Windows 11 showed playback timestamp avgGap stuck at ~45ms (≈22fps) on `fhd_30fps_2h.mkv`. Two confirmed root causes:

1. **Windows.Forms.Timer precision limit:** `timerPlayback.Interval = 33` was being rounded by the OS scheduler to ~45ms. `timeBeginPeriod` does not affect WM_TIMER granularity.
2. **`lastFrameTime = currentTime` drift bug:** When Timer fired 47ms after `lastFrameTime`, `framesToMove = (int)(47/33.3) = 1`, advancing 33.3ms of frame time but discarding the 14ms remainder by overwriting `lastFrameTime` to `currentTime`. The 14ms loss accumulated every tick, locking playback at ~70% of real-time (≈22fps).

GS인증 1등급 결함 0건 기준 (CLAUDE.md) 에 따라 hot-path High-severity 결함으로 closure 필요.

## Changes (Atomic, Single File, 2 lines)

### Forms/MainForm.cs

**Change 1 — `btnPlay_Click` (StartPlayback) — Timer Interval:**

```diff
-                    timerPlayback.Interval = 33;
+                    timerPlayback.Interval = 8;
                     timerPlayback.Start();
```

WM_TIMER fires at ~8ms granularity (sufficient to avoid 33→45ms OS rounding). The internal advance decision `elapsedMs >= msPerFrame` (33.3ms) is unchanged — Interval 8ms acts as a polling step, not an advance step. framesToMove=0 ticks are near-free (one `DateTime.Now.Ticks` read + subtraction + integer divide).

**Change 2 — `timerPlayback_Tick` — Drift-free `lastFrameTime` update:**

```diff
             if (framesToMove > 0)
             {
                 int nextFrame = Math.Min(_videoService.TotalFrames - 1, _videoService.CurrentFrameIndex + framesToMove);
                 LoadFrame(nextFrame);
-                lastFrameTime = currentTime;
+                lastFrameTime += (long)(framesToMove * msPerFrame);
```

The accumulator approach advances `lastFrameTime` by exactly the ideal time consumed (`framesToMove * msPerFrame`), preserving the unused remainder of `elapsedMs` for the next tick. This produces catch-up semantics: if Timer fires late, next tick can advance 2 frames to recover. Drift is eliminated permanently.

**Preserved (untouched):**

- `if (framesToMove > 0)` guard (line 1076) — guarantees `framesToMove * msPerFrame > 0`, blocking 0 accumulation.
