## Decisions Made

- **ProcessCmdKey intercept**: `MainForm_KeyDown` 의 Tab 분기는 주석으로 보존 (역사적 참조), 실제 동작은 ProcessCmdKey 로 이관. 영상 미로드/BBOX 0건 상황에서도 Tab 을 조용히 흡수하여 버튼 focus 이동 방지.
- **PushModifyBoxUndo 공용 헬퍼**: 3 경로 (W/A/S/D 키보드, mouse drag, mouse resize) 모두 동 헬퍼 경유. `currentState, originalRect, originalLabel, originalObjectId` 시그니처로 Undo 계약의 3필드(`OriginalRectangle/OriginalLabel/OriginalObjectId`) 강제 세팅. ChangeBoxIdOnly (Wave 2) 와 동일 계약 유지.
- **per-keystroke W/A/S/D undo (coalesce 없음)**: 각 키 입력당 독립 ModifyBox 엔트리. 사용자 기대(Ctrl+Z 한 번 = W 한 번 취소) 일치. 스택 절약 < 예측 가능성.
- **drag/resize no-op 차등 체크**: `rect != original` 비교로 no-op MouseUp (그냥 클릭) 시 스택 오염 방지. `originalDragRect` 필드로 drag-start 상태 보존.
- **Undo AddBox / Redo RemoveBox 2단계 매칭**:
  - Stage 1 `ReferenceEquals(b, action.Box)` fast-path — clone round-trip 없는 일반 시나리오에서 정확히 한 박스 매치.
  - Stage 2 `FrameIndex + Label + Id + Rectangle` 값 매칭 — AddBox→Remove→Undo Remove→Undo Add 시퀀스에서 orphaned live ref 대응.
  - Rectangle 이 완벽한 키는 아니지만 "두 박스를 정확히 같은 좌표로" 는 실사용 시나리오에서 극히 드물며 시각적 구분 불가이므로 안전한 disambiguator 로 간주.
- **OnboardingForm Shown 이벤트 기반**: Form Load 가 아닌 OnShown 에서 트리거 — Form 레이아웃 완료 후 ShowDialog → CenterParent 동작 정상. OnShown 예외는 `Log.Warning` 으로 삼켜 앱 동작 보호.
- **"다시 보지 않기" 체크 + "완료" 조합만 MarkOnboardingShown**: 단순 X / ESC / Cancel 은 플래그 저장 안 함 — 재실행 시 다시 표시.
- **SettingsService JSON.NET 사용**: System.Text.Json 대신 이미 프로젝트 의존성인 Newtonsoft.Json 선택 — 추가 deps 없이 `Formatting.Indented` 직렬화.
- **메시지 한국어화 원인/해결 포함**: 원 plan 지침 "사용자 친화적 (원인 + 가능한 해결)" 준수 — `"비디오 파일을 열 수 없습니다.\n\n원인: ...\n해결 방법: MP4(H.264) 등 지원 형식..."`.
- **Waypoint 단일 선택 UX 보존**: 기존 `"선택한 {type} Waypoint"` + 타이틀 유지, 다중만 신규 `"선택된 N개의 Waypoint"` + 타이틀 `"Waypoint 삭제"`.
- **UAT checkpoint 원칙 재확인**: "조치완료" 선언은 UAT 검증 없이 신뢰 금지. DEFECTS-INBOX 의 status 컬럼이 "조치완료" 여도 checkpoint:human-verify 가 없다면 실제 동작 불확실.

**3. [Rule 1 - Bug] DF-1-07 Undo AddBox / Redo RemoveBox 참조 동일성 실패 (clone round-trip)**
- **Found during:** Task 5 (UAT 2차 재검증 FAIL, 6단계 시나리오)
- **Issue:** UAT 시나리오 "A 생성 → A 이동 → A 리사이즈 → B 생성 → B 삭제 → C 생성 → Ctrl+Z 반복" 에서 Ctrl+Z 3번째 (Undo AddBox B) 에서 B 가 화면에 남음. 원인 분석: `Undo()` `case UndoActionType.AddBox:` 가 `boundingBoxes.Remove(action.Box)` 로 **참조 동일성 기반** 제거. `List<T>.Remove` 는 `Equals` 미override 타입에 대해 reference equality 사용. 시퀀스 trace:
  - Step D (B 생성): AddBox action.Box = B_live (참조 그대로 저장)
  - Step E (B 삭제): RemoveBox action.Box = `CloneBoundingBox(B_live)` = B_clone. `boundingBoxes.Remove(B_live)` 실행.
  - Ctrl+Z 2 (Undo RemoveBox): `boundingBoxes.Add(B_clone)` — 리스트에 B_clone 이 들어감.
  - Ctrl+Z 3 (Undo AddBox B): `boundingBoxes.Remove(B_live)` 시도 — 리스트에는 B_clone 뿐 → 참조 불일치로 `Remove` 가 `false` 반환 후 조용히 실패.
- **Fix:** `Undo()` AddBox 및 `Redo()` RemoveBox 케이스를 값 기반 predicate 매칭으로 교체 (MainForm.cs:2400-2458). 매칭 키 = `FrameIndex + Label + GetBoxId(b)` (기존 ModifyBox 케이스와 동일 패턴). Rectangle 은 의도적 제외 — ModifyBox 가 live 박스의 Rectangle 을 in-place 로 변경하므로 매칭 키로 신뢰 불가. `selectedBox == action.Box` 비교도 발견된 live ref (`toRemove`) 로 교체.
- **Files modified:** Forms/MainForm.cs (Undo/Redo 2 케이스), .planning/phases/05.6-결함수정/05.6-CONTEXT.md (D-13 "수정 5-C")

**4. [Rule 1 - Bug] DF-1-07 매칭 키 ambiguity 회귀 (같은 클래스 + 같은 기본 Id)**
