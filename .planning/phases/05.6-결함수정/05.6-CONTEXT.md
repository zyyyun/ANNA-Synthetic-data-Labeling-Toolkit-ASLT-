## Implementation Decisions

### ID 관리 서브시스템 재설계 (locked, user-confirmed)

- **D-01 Ctrl+N 신규 구현**: Exit 프레임에서 Ctrl+N 누르면 Exit 프레임의 선택된/모든 BBOX ID를 Entry 프레임의 ID와 자동 일치시킨다. 이미 오류 메시지(`MainForm.cs:996`)에서 "해결 방법"으로 안내되는 문구 그대로 실구현.
- **D-02 선택 박스 단위 ID 변경 (NEW-05/07 수정)**: `ChangeBoxIdWithinWaypoint`를 "선택된 개별 BBOX에만 ID 변경 적용"으로 수정. Waypoint가 같더라도 다른 박스에 전파 금지. 사용자 명시: "waypoint가 같더라도 선택한 개별 bbox의 id만 변경될 수 있도록 설정".
- **D-03 Event 클래스 ID 불일치 경고 추가 (NEW-02)**: `MainForm.cs:987-991` 의 Entry-Exit ID 검증 로직에 Event 클래스 추가. Person/Vehicle과 동일 포맷 경고 메시지.
- **D-04 포커스 가드 재설계 (NEW-04)**: `MainForm_KeyDown:2332-2336`의 TextBox/ComboBox 가드가 `Ctrl+1~0 / Ctrl+N` 같은 단축키를 먹지 않도록 예외 추가. ListView 포커스에서도 단축키 동작.
- **D-05 Person 사이드바 ID 변경 UI (NEW-06)**: Person Labels 영역에 Vehicle/Event와 동일 수준의 ID 편집 컨트롤 추가. (범위 확인 필요 — 기존 Vehicle/Event UI 패턴 복제)
- **D-06 BBOX 생성 시 ID 승계 (NEW-01)**: BBOX 생성 직후 `currentAssignedId`가 있으면 그걸 사용; 없으면 현 프레임에서 같은 클래스의 선택된 박스가 있으면 그 ID 승계. Exit 프레임 새 BBOX 생성 시 Entry 프레임의 ID 사용 로직 추가.

### BBOX/Waypoint/JSON 정합성 (locked, QA 요구 반영)

- **D-07 Waypoint 동반 삭제 프롬프트 (DF-1-03)** — canonical UI string (LOCKED): `"Waypoint 구간 BBOX가 전부 삭제됩니다. 해당 Waypoint도 함께 삭제하시겠습니까?"` (Title: `"Waypoint 삭제"`, `MessageBoxButtons.YesNo`, `Warning` icon). ### 사용성

- **D-11 크래시 가드 (DF-1-13)**: `btnEntry_Click`와 `btnExit_Click` 진입부에 `_videoService.IsVideoLoaded` 체크 추가. 미로드 시 한국어 안내 메시지 "영상을 먼저 로드해 주십시오." 표시 후 return. 기존 `TimeSpan.FromSeconds(fps NaN)` 호출 경로 원천 차단. `Keys.E / Keys.X` 단축키도 동일 가드.
- **D-12 Tab 순환 (DF-1-06)** — **업데이트 2026-04-24 (05.6-04 실행 중 발견 + 수정)**: 기존 `MainForm_KeyDown` 의 Tab 분기(`CycleSelection(e.Shift)`)는 **도달 불가능 dead code** 였음. WinForms `ProcessDialogKey` 가 Tab 을 버튼 traversal 용도로 먼저 소비하여 `KeyDown` 이벤트 자체가 발생하지 않아 BBOX 순환이 실제로 동작하지 않았다. **수정**: `ProcessCmdKey` override (line ~2479) 에 Tab/Shift+Tab 가로채기 블록을 추가 — `ProcessDialogKey` 이전 단계에서 `return true` 로 버튼 traversal 차단, 영상 로드 + BBOX ≥ 1 조건 충족 시 `CycleSelection(reverse: shift)` 호출. 영상 미로드/BBOX 0건 상황에서도 Tab 은 조용히 흡수 (버튼 focus 이동 방지). `MainForm_KeyDown` 의 구 Tab 분기는 주석으로 보존. 사용자취급설명서 명문화는 **Phase 6-문서화로 이월**.
- **D-13 Ctrl+Z Undo 검증 (DF-1-07)** — **업데이트 2026-04-24 (05.6-04 실행 중 발견 + 수정)**: UAT 시도 1차 FAIL. 기존 "직전 작업 단위 Undo 수정 선언" 은 사실상 미완성이었음. 발견된 결함 3종:
  1. **W/A/S/D 키보드 이동 경로** (`MainForm_KeyDown` line ~2683) 에 `AddUndoAction` 호출 자체가 없어 undo 스택에 전혀 기록되지 않았다.
  2. **마우스 drag 이동 MouseUp 경로** (`pictureBoxVideo_MouseUp` else-if `isDragging` 블록) 도 undo 푸시가 없어 기록되지 않았다.
  3. **마우스 resize MouseUp 경로** 는 `AddUndoAction` 호출은 있었으나 `UndoAction.Box.Rectangle = originalResizeRect` 만 세팅하고 `OriginalRectangle/OriginalLabel/OriginalObjectId` 3 필드를 놓쳤다. `Undo()` 핸들러 계약 위반 — 결과적으로 resize Ctrl+Z 가 선택 박스의 Rectangle 을 `(0,0,0,0)`, Label 을 `null`, Id 를 `0` 으로 덮어써 박스를 "보이지 않는 상태" 로 파괴했다.
  **수정 5-B**: 공용 헬퍼 `PushModifyBoxUndo(currentState, originalRect, originalLabel, originalObjectId)` 신설 (`MainForm.cs:2388`). 세 경로 모두 동 헬퍼 경유. W/A/S/D 는 per-keystroke 푸시 (coalesce 없음). Mouse drag/resize 는 `rect != original` 차등 체크로 no-op 클릭 시 스택 오염 방지. `originalDragRect` 필드 (`MainForm.cs:~71`) 를 MouseDown 3개 drag-start 지점에서 스냅샷.

  **추가 발견 (5-B UAT 2차 재검증 FAIL 1건)**: "A 생성 → A 이동 → A 리사이즈 → B 생성 → B 삭제 → C 생성" 6단계 UAT 에서 Ctrl+Z 3번째 (Undo AddBox B) 가 동작하지 않음 — B 가 화면에 남음.
  4. **Undo AddBox / Redo RemoveBox 참조 불일치 버그**: `Undo()` `case UndoActionType.AddBox:` 가 `boundingBoxes.Remove(action.Box)` 로 **참조 동일성 기반** 제거를 수행. `List<T>.Remove` 는 `Equals` 미override 타입에 대해 reference equality 를 사용하므로, clone round-trip 을 거친 시나리오에서 실패:
     - Step D (B 생성): AddBox undo action 의 `Box` = B_live (참조 그대로 저장, line 1823 `Box = drawingBox`).
     - Step E (B 삭제): RemoveBox undo action 의 `Box` = `CloneBoundingBox(B_live)` = B_clone. `boundingBoxes.Remove(B_live)` 실행.
     - Ctrl+Z 2 (Undo RemoveBox): `boundingBoxes.Add(B_clone)` — 리스트에 **B_clone** 이 들어감.
     - Ctrl+Z 3 (Undo AddBox B): `boundingBoxes.Remove(B_live)` 시도 — 리스트에는 B_clone 밖에 없음, 참조 불일치로 `Remove` 가 `false` 반환하고 조용히 실패. B 가 화면에 남음.
  **수정 5-C**: `Undo()` AddBox 및 `Redo()` RemoveBox 케이스를 **값 기반 predicate 매칭** 으로 교체 (`MainForm.cs:2400-2458`). 매칭 키 = `FrameIndex + Label + GetBoxId(b)` (기존 ModifyBox 케이스와 동일 패턴). **Rectangle 은 의도적으로 제외** — ModifyBox 가 live 박스의 Rectangle 을 in-place 로 변경하므로 매칭 키로 신뢰 불가. `selectedBox == action.Box` 비교도 발견된 live ref (`toRemove`) 로 교체. 사용자취급설명서 명문화는 Phase 6-문서화 이월.

  **추가 발견 (5-C UAT 3차 재검증 FAIL 1건, commit `f7ecd12` 회귀)**: 동일 6단계 UAT — A/B/C 모두 Person 클래스 기본 Id=1 로 생성. Ctrl+Z 1번째 (Undo AddBox C) 에서 화면에서 **C 가 아닌 A 가 사라짐**.
  5. **매칭 키 모호성 버그**: 5-C 의 매칭 키 `FrameIndex + Label + GetBoxId(b)` 는 **같은 클래스·같은 기본 Id** 를 가진 모든 박스에 대해 성립. A/B/C 가 모두 `Frame=F, Label="person", Id=1` 이면 `FirstOrDefault` 가 삽입 순서상 최초 항목(A)을 반환 → 잘못된 박스 제거.
  **수정 5-D**: `Undo()` AddBox 및 `Redo()` RemoveBox 를 **2 단계 매칭** 으로 재교체 (`MainForm.cs:2400-2461`).
  - **Stage 1 (fast-path)**: `ReferenceEquals(b, action.Box)` — AddBox action.Box 는 live 참조(5-C 에서도 변경 없음)이므로 clone round-trip 이 없는 일반 시나리오에서 정확히 한 박스에 매치.
  - **Stage 2 (fallback, null-coalesce)**: `FrameIndex + Label + GetBoxId(b) + Rectangle` — AddBox→Remove→Undo Remove→Undo Add 시퀀스에서 리스트의 박스는 Remove 시점 clone 이며 action.Box 는 orphaned live 참조. Rectangle 은 AddBox 이후 clone 시점까지 ModifyBox 가 끼어들지 않는 한(언더로 unwind 되므로 순서상 불가) 동일 값이므로 안전한 disambiguator.
  Rectangle 이 완벽한 키는 아니지만 "두 박스를 정확히 같은 좌표로 그린 상태" 는 실사용 시나리오에서 발생 가능성 극히 낮음(두 박스가 시각적으로 구분 불가). 사용자취급설명서 명문화는 Phase 6-문서화 이월.
- **D-14 온보딩 가이드 UI (DF-1-11)**: 최초 실행 시 "동영상 선택 → BBOX 생성 → Entry/Exit 설정" 3단계 가이드 팝업. 이후 "다시 보지 않기" 체크 시 %LOCALAPPDATA%\ANNA\ASLT\settings.json 같은 곳에 플래그 저장. 구현 상세는 planner 재량.
- **D-15 영상 로드 메시지 한국어화 (DF-1-14)**: "Failed to open video file." → "비디오 파일을 열 수 없습니다." 등. `Services/VideoService.cs` `LoadVideoAsync` 예외 메시지 전부 검토.
- **D-16 Waypoint 일괄 삭제 (DF-1-18)**: listViewPerson/Vehicle/EventWaypoints 의 다중 선택 상태에서 "선택한 Waypoint 삭제" 버튼이 전체 선택에 일괄 적용. 이미 `btnDeleteSelectedWaypoint_Click` 핸들러 존재 확인됨 (`MainForm.cs:2450`) — 현재 단일만 처리하는지 검증 후 다중 대응.
