### D-17b: HMAC 무결성 체인

- **Wire format**: `{timestamp} [{level}] {message}|prev_hmac={prev_hex}|hmac={hmac_hex}`
- **GENESIS sentinel**: 파일 시스템에 이전 로그 파일이 0 개이고 이 파일이 첫 라인일 때만 prev=GENESIS
- **Cross-day chain**: 프로세스 내 자연 rollover 는 in-memory `_lastHmac` 유지 (어제 마지막 hmac 이 오늘 첫 라인의 prev) / 앱 재시작 후 날짜가 다를 때 최근 prior-day 파일의 마지막 hmac 을 fallback 으로 이어받기
- **Recovery**: 같은 날 내 앱 재시작 시 기존 파일의 마지막 hmac 을 디스크에서 읽어 체인 계속
- **Thread safety**: Sink `Emit` 내부 `lock(_lock)` — 단일 프로세스 멀티스레드 안전. 멀티프로세스 동시 쓰기는 스코프 밖 (WinForms 단일 인스턴스 전제).

**참고: CLI 스탠드얼론 유틸은 생성하지 않았다.** DF-1-17 level c 요구 범위는 검증 API 제공까지이며, 별도 EXE 배포는 요청되지 않았다. 운영 시 외부 감사 도구에서 다음 snippet 로 호출 가능:

```csharp
// 별도 companion 콘솔 프로젝트에서 ASLTv1.dll 참조:
using ASLTv1.Services;

byte[] key = HmacKeyProvider.GetOrCreateKey();
var result = LogIntegrityVerifier.VerifyLogFile(
    logFilePath: @"C:\Users\{user}\AppData\Local\ANNA\ASLT\logs\AOLT-2026-04-24.log",
    key: key,
    initialPrevHmac: "GENESIS");  // 첫 파일이면 GENESIS, 아니면 전날 파일의 마지막 hmac

Console.WriteLine($"IsValid: {result.IsValid}");
Console.WriteLine($"Total lines: {result.TotalLines}, Verified: {result.VerifiedLines}");
if (!result.IsValid)
    Console.WriteLine($"Tampered line {result.TamperedLine}: {result.Reason}");
```

Python 도구 사용자의 경우 위의 Python 스크립트를 그대로 활용 (C# 코드와 byte-for-byte parity 확인됨).

## Decisions Made

### Architecture

- **Serilog ILogEventSink 경유 HMAC 부착** — 기존 File sink 완전 대체. 두 Sink 병렬은 이중 기록 + HMAC 비일치로 부적절.
- **30일 보존은 CleanupOldLogs 로 재구현** — 커스텀 Sink 는 Serilog 의 rollingInterval/retainedFileCountLimit 옵션을 상속하지 않음. `LogService.Initialize` 시작 시점 1회 정리. 30일 보존 계약 (D-17c) 유지.
- **GENESIS sentinel 는 파일 + 디렉터리 둘 다 부재일 때만** — prior-day 파일이 있으면 그 파일의 마지막 hmac 을 prev 로 사용 (CONTEXT D-17 "일별 로테이션 경계" 규약 준수).

### Key Management

- **`out bool wasCreated` 플래그로 logger-ready ordering 해결** (advisor 가이드 1번) — HmacKeyProvider.GetOrCreateKey 는 Log.* 금지. Logger 가 초기화 완료된 후 LogService.Initialize 가 직접 `Log.Information("[AUDIT] HMAC 키 생성: ...")` 기록.
- **FileAttributes.Hidden best-effort** — try/catch 로 일부 파일 시스템 미지원 대응 (FAT/ReFS 등 일부 속성 비지원 환경).

### Audit Signal Quality

- **BBOX Create 는 MouseUp 1지점만** (advisor 가이드 3번) — 프로그래밍 전파 경로 4개 (line 502 track continuation, 2480/2504/2537 Event/P/V propagation) 와 Undo/Redo 2개 (line 2426/2442) 은 의도적 제외. 이유: 사용자가 한 번 그린 박스가 Exit 프레임 전파로 50 프레임에 걸쳐 생성 시 감사 라인 폭증. 감사 로그는 사용자 의사결정 기록이 주 목적이지 내부 전파 메커니즘 기록이 아님.
- **BBOX Delete 는 Delete/G + 사이드바 2지점만** — btnDeleteSelectedWaypoint_Click 의 foreach 박스 삭제는 Waypoint 삭제 감사 이벤트가 상위 맥락으로 흡수 (Waypoint 한 번 삭제 = n 박스 삭제 이지만 감사 라인은 Waypoint 1개만 기록되는 것이 자연스러움).
- **Waypoint Create 는 Entry-Exit 성공 경로 3 foreach (P/V/E)** — 이미 overlapping skip 분기로 필터링되어 신규 생성에만 로그.
- **Waypoint Delete 는 사용자 주도 2지점** — btnDeleteSelectedWaypoint_Click 일괄 + PromptWaypointDeletionIfEmpty Yes 분기.

### Cross-day Continuity

- **3단계 분기** (`_lastFilePath` 비교 후):
  1. 파일 존재 → `ReadLastHmac(filePath)` 복구 (same-day restart)
  2. `_lastFilePath == null` && 파일 부재 → `TryReadLastHmacOfMostRecentPriorFile()` 시도 (day-N first boot with day-(N-1) file existing)
  3. else → `_lastHmac` 유지 (process-internal rollover)
- GENESIS 는 **오직** 모든 prior 상태가 부재일 때만 (true first run + empty log dir).

### Defensive Emit

- **try/catch around AppendAllText** — IOException (파일 락) / UnauthorizedAccessException (권한) 시 Debug.WriteLine fallback + `_lastHmac` 미업데이트. 다음 Emit 재시도.
- **외부 try/catch** — Emit 전체를 감싸 최후방어. `Log.Fatal` → Sink → 예외 → 글로벌 예외 핸들러 → `Log.Fatal` 재귀 루프 방지.

### Legacy Migration

- **Wave 3 raw `Log.Information("[AUDIT] JSON 파일 삭제: ...")` 2 호출 → `LogService.AuditJsonDelete(path)` 치환** — 동일 문자열 포맷 ("[AUDIT] JSON 파일 삭제: {Path}") 유지. 래퍼 경유로 감사 이벤트 규약 일관성 확보. 기존 감사 문자열을 grep 하는 외부 도구도 호환.
